# 02 数据库模块审阅（`src/db`）

## 当前结论

数据库主运行链已经完成从 ORM 到 pooled PostgreSQL + 显式 SQL 的收口：

- `src/db/pool.py`
  - 提供 Python 版 `Db { pool }`
- `src/db/database.py`
  - 统一初始化共享连接池
- `src/db/sql_repo.py`
  - 承担 task / completions / eval / checker / scores 的显式 SQL 读写
- `src/db/eval_db_service.py`
  - 只保留业务编排，不再依赖 ORM session/repository

## 已收敛的关键问题

### DB-1. completion attempt 索引已改为严格校验
- 位置：`src/db/sql_repo.py`
- 现状：completion 写入通过 `strict_nonneg_int(...)` 校验 `sample_index / repeat_index / pass_index`，不再把非法值兜底成 `0`。

### DB-2. eval/checker/score 已使用唯一键 upsert 语义
- 位置：`src/db/sql_repo.py`
- 现状：
  - `eval` 通过 `ON CONFLICT (completions_id) DO UPDATE`
  - `checker` 通过 `ON CONFLICT (completions_id) DO UPDATE`
  - `scores` 通过 `ON CONFLICT (task_id) DO UPDATE`

### DB-3. benchmark/model/task identity 已转为 SQL get-or-create / identity lookup
- 位置：`src/db/sql_repo.py`
- 现状：benchmark/model 走唯一键维度查找与插入；task resume / rerun 使用统一 identity lookup。

## 还值得继续关注的点

### DB-4. `EvalDbService` 仍偏胖
- 位置：`src/db/eval_db_service.py`
- 问题：service 仍同时承担 resume 判定、payload 清洗、上下文重建、导出读取。
- 建议：后续按“resume / writes / reads / export”继续拆薄。

### DB-5. DB writer 仍是线程，而不是独立 writer 进程
- 位置：`src/db/async_writer.py`
- 问题：当前只是线程内异步刷写，尚未形成真正的独立 DB writer process。
- 建议：如果继续向 orchestrator 主干推进，应单独做 writer 进程边界。
